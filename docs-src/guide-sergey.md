# Тест меша MeshCore перед фестом — пошаговая инструкция 🐢

**Три части: что делать дома в нодах → что делать дома с флиппером → что делать в поле.**
Ноута и телефона в поле нет: всё крутится с одного флиппера.

> 💡 Это версия под **флиппер**: дома методика та же, что была с телефонами, —
> поменялся только инструмент оператора. У Марка остаётся телефон:
> [гайд Марка](guide-mark.html).

> 🛶 **Полевая часть переписана под сплав по реке.** Части 1–2 (дома) не
> изменились. Часть 3 у вас с Марком **одна и та же** — сценарии совместные,
> и в его гайде тот же текст слово в слово.

## Расклад железа (5 нод)

| # | Нода | Роль | Прошивка | GPS+экран | Кто рулит |
|---|------|------|----------|-----------|-----------|
| 1 | **V4** (ESP32-S3) | 🎒 **ROVER-S** — логгер A | наша UART-companion | ✅ | Флиппер, LPUART (15/16) |
| 2 | **T114** | 🎒 **POCKET-S** — логгер B | наша UART-companion | ✅ | Флиппер, USART (13/14) |
| 3 | **T114** | 🏠 **BASE** — якорь, высоко | стоковый `repeater` | — | по радио / USB |
| 4 | **T114** | 📡 **REP1** — скаут-ретранслятор | стоковый `repeater` | ✅ | по радио / USB |
| 5 | **T114** | 📡 **REP2** — ретранслятор | стоковый `repeater` | — | по радио / USB |

**Почему теперь так.** Раньше V4 был единственным логгером, потому что только у него
есть WiFi, а телефон умел говорить с нодой лишь по TCP. Флипперу WiFi не нужен —
он говорит с обеими нодами по проводу. Отсюда два следствия:

- **V4 больше не поднимает WiFi-точку.** Прошивка `companion_radio_wifi` не нужна,
  и это заметно меньше жора — повербанк перестаёт быть обязательным.
- **Обе ноды слушают эфир одновременно**, в одной точке, в одну секунду. Это даёт
  честное **A/B-сравнение V4 против T114** — раньше так было нельзя.

GPS ставим туда, где он реально нужен: на обоих роверов и на REP1 — его ты будешь
переставлять. База стоит на известной точке, ей координаты пропишем руками.

# ЧАСТЬ 1. ДОМА — В НОДАХ 🔧

Делается один раз, за столом, с компьютера по USB.

> ⚠️ **Разнеси ноды на 2–3 метра, пока настраиваешь.** Пять передатчиков по 20+ dBm
> в 20 см друг от друга перегружают приёмники, и ты словишь странные «глюки связи»,
> которых в поле не будет.

### 1.0 Подготовка: CLI на компьютере (Windows)

`pipx` ставить **не надо** — обычного `pip` хватает.

```powershell
py --version
py -m pip install --user meshcore-cli
meshcli -v
```

Если `meshcli` не находится — папка `Scripts` не попала в PATH. Не воюй, запускай
через модуль:

```powershell
py -m meshcore_cli -v
```

**Ищем COM-порт ноды:**

```powershell
meshcli -l
```

- V4 не появился → доставь драйвер **CP210x**.
- T114 на nRF52840 определяется сам.

> ⚠️ Если домашняя папка с кириллицей и полезут ошибки кодировки — сделай venv по
> латинскому пути: `py -m venv C:\mesh\venv`, затем `C:\mesh\venv\Scripts\activate`.

### 1.1 Прошить

Все пять — с нашего флешера: [hleserg.github.io/meshcore-node-fw](https://hleserg.github.io/meshcore-node-fw)

| Ноды | Образ |
|---|---|
| **Два ровера** (V4 + T114) | **UART-companion** — с автодетектом флиппера |
| **BASE, REP1, REP2** | `repeater` |

> 🚨 **Роверам нужна именно наша сборка с автодетектом.** Стоковый companion отдаёт
> протокол только в USB-CDC или BLE, а флиппер умеет исключительно аппаратный UART
> и до него не достучится. На T114 без этого нода будет молчать намертво.

> 💡 Наша сборка **не отнимает** у ноды радио: она смотрит на пин DETECT при старте
> и выбирает, кем работать. Провода нет — поднимает WiFi/BLE как стоковая.

### 1.2 Поставить одинаковое радио на ВСЕХ ПЯТИ

Самый частый и самый обидный косяк: если freq/BW/SF/CR не совпадают — ноды физически
не слышат друг друга, и ты полдня ищешь несуществующую поломку.

```
freq = 869.525 МГц   BW = 250 кГц   SF = 11   CR = 5
```

Через CLI, подставь свой COM-порт:

```powershell
meshcli -s COM5 set radio 869.525,250,11,5
meshcli -s COM5 set name ROVER-S
```

Репитеры настраиваются своим CLI по USB: `meshcli -s COM5 -r`, дальше `set ...`.

> 💡 Синтаксис `set` немного разнится между версиями прошивки. Не гадай — набери
> **`set help`** в CLI ноды и посмотри список.

> ✅ **Роверам радио можно ставить и с флиппера** — это быстрее и без компьютера.
> Как: раздел «Пресеты» ниже, пункт 2.3.

### 1.3 Дать имена

Твои пять: `BASE`, `REP1`, `REP2`, `ROVER-S`, `POCKET-S`.
Ровно этими именами ты потом будешь пинговать, и они попадут в CSV.
Наклей стикеры на корпуса — в поле спасает.

> 🤝 **Мы объединяемся с Марком, в сети будет 8 нод.** Он настраивает свои три по
> [отдельной инструкции](guide-mark.html): `ROVER-M`, `POCKET-M` и `REP3`.
> **Радио-настройки обязаны совпадать** у обоих комплектов до последней цифры.
> Контактами обменяйтесь дома: `export_contact` выдаёт URI, его можно переслать.

### 1.4 Сменить пароли на трёх репитерах

У репитера есть админ-CLI по радио, и дефолтный пароль общеизвестен. Ставь свой —
иначе любой желающий на фесте перенастроит тебе сеть. Запиши пароли в телефон.

### 1.5 Синхронизировать часы

```powershell
meshcli -s COM5 clock sync
```

Без этого метки времени разъедутся, и сопоставить твои данные с данными Марка
не выйдет.

### 1.6 Прописать координаты базе

BASE поставишь в известную точку — вбей её широту/долготу заранее (`set lat`, `set lon`).

### 1.7 Познакомить ноды между собой

Включи все пять рядом (помним про 2–3 метра) и дай каждой отправить advert.
Проверь, что каждая видит остальных:

```powershell
meshcli -s COM5 contacts
```

**Зачем:** в поле ты не хочешь стоять и ждать, пока ноды обнаружат друг друга.

### 1.8 Зарядить и собрать

Всё под завязку. **V4 без WiFi-точки жрёт заметно меньше, чем раньше** — повербанк
теперь страховка, а не обязательное условие.

**✅ Чек-лист части 1:** прошито 5/5 (роверы — нашей UART-сборкой!) · радио одинаковое ·
имена заданы · пароли сменены · часы синхронизированы · координаты базы вбиты ·
контакты видны · всё заряжено.

# ЧАСТЬ 2. ДОМА — СОБРАТЬ РИГ 🔌

Termux, WiFi-точки, IP-адресов и TCP больше нет. Вместо них — четыре провода на ноду,
одно меню и одно правило про перезагрузку.

### 2.1 Распайка и два кабеля

{{svg:wiring}} | Два непересекающихся блока на гребёнке флиппера и два кабеля к ним. Со стороны ноды разъём одинаковый, поэтому кабели различаются только порядком на флиппере.

Со стороны **ноды** разъём всегда один и тот же: **GND · DETECT · RX · TX**.
Со стороны **флиппера** блоки зеркальные, поэтому кабеля два.

| Провод | Кабель A → блок A | Кабель B → блок B |
|---|---|---|
| GND | пин **11** | пин **18** |
| DETECT | пин **12** | пин **17** |
| RX ноды | пин **13** (TX флиппера) | пин **15** (TX флиппера) |
| TX ноды | пин **14** (RX флиппера) | пин **16** (RX флиппера) |

Куда паять на самих платах — **по номерам на хедере**, а не по названиям GPIO:

| Сигнал | T114 — хедер P1 | V4 — хедер J2 |
|---|---|---|
| GND | **P1·4** | **J2·1** |
| DETECT | **P1·8** = GPIO33 | **J2·12** = GPIO33 |
| RX | **P1·12** = GPIO9 | **J2·13** = GPIO47 |
| TX | **P1·13** = GPIO10 | **J2·14** = GPIO48 |

> 💡 **Мнемоника:** DETECT на обеих платах — **GPIO33**. Одно число на две платы,
> его легко держать в голове с паяльником в руке.

> ✅ У обеих плат всё на **одном** хедере — ничего тянуть через плату не нужно.

> 🚨 **Питание нод — своё, не от флиппера.** Только GND, DETECT и две линии данных.
> Флиппер не рассчитан тянуть LoRa-ноду через передачу, а просадка питания
> посреди прогулки стоит тебе всей сессии.

> ⚠️ **TX↔RX перекрёстно.** Выход одной стороны идёт во вход другой. Соединить
> TX с TX — самая частая причина «нода молчит», и выглядит она в точности как
> «нода сдохла».

### 2.2 Detect — как нода понимает, кто ей рулит

Наша прошивка смотрит на пин DETECT **в момент старта ноды** и по нему выбирает,
с кем работать:

| DETECT | Что делает нода |
|---|---|
| **HIGH** (провод воткнут, приложение открыто) | остаётся в serial-режиме, рулит флиппер, своё радио не поднимает |
| **LOW** (провода нет) | поднимает своё радио — V4 точку WiFi, T114 BLE — и работаешь телефоном |

Флиппер держит пины 12 и 17 в HIGH всё время, пока приложение открыто, и отпускает
их при выходе.

> 🚨 **Правило, которое надо запомнить целиком:**
> **нужен провод — воткнул и перезагрузил ноду; нужен телефон — вынул и перезагрузил ноду.**
> Горячего переключения нет и не будет: нода спрашивает линию ровно один раз, на старте.

> ⚠️ **Выключи Debug на флиппере** перед работой: Settings → System → Debug: **OFF**.
> Пин 12 — это PA13, он же SWDIO, и в режиме отладки им владеет отладочный блок.
> Приложение честно предупредит об этом на экране Connect, но лучше просто выключить.

> ✅ Радио на нодах **не выключается насовсем**. Оно просто не поднимается, пока
> воткнут флиппер. Вынул провод, ребутнул — и телефон снова видит ноду как раньше.

### 2.3 Первое подключение

1. Воткни ноду, включи флиппер.
2. **Apps → GPIO → MeshCore Config**.
3. Выбери **Connect**, нажми **OK**.

{{svg:screen-connect}} | Экран после успешного подключения: модель ноды, версия прошивки, имя и текущее радио.

Успех выглядит так: на экране модель (`Heltec T114` или `Heltec V4`), версия прошивки,
имя ноды и радио-настройки.

Если вместо этого написано **«No reply from the node»** — иди в раздел «Грабли».

> 💡 Пункт **Serial log** в главном меню показывает сырой обмен байтами. Если
> подключение не идёт — открой его: пусто = до флиппера вообще ничего не дошло,
> мусор = не тот бод или перепутаны TX/RX.

### 2.4 Пресеты — поставить радио с флиппера

Вместо того чтобы гонять `set radio` по USB на каждую ноду, можно применить пресет.

1. **Presets** в главном меню.
2. Первым в списке — встроенный **City/daily** (868.731 / 62.5 / SF7 / CR7).
   Он вшит в приложение и есть всегда, даже без SD-карты.
3. Свои пресеты клади на SD в `/ext/apps_data/meshcore_cfg/presets/*.json`.
   Для феста нужен вот такой:

```json
{
  "name": "Fest-869",
  "freq_mhz": 869.525,
  "bw_khz": 250,
  "sf": 11,
  "cr": 5,
  "path_hash_bytes": 1
}
```

4. Выбрал пресет → **OK** → приложение шлёт настройки и **перечитывает их обратно**,
   показывая галочки по каждому полю. Галочка означает «нода подтвердила», а не
   «мы отправили».

> ✅ **Save from node** делает обратное: считывает конфиг воткнутой ноды в новый
> `*.json`. Настроил одну ноду как надо — сохрани и клонируй на остальные.

### 2.5 Генеральная репетиция дома (не пропускай!)

1. Воткни обе ноды, зайди в **Logger**, нажми **OK**.
2. Походи по квартире 2–3 минуты.
3. Вернись, нажми **Back** — логгер закроет файлы.
4. Вынь SD, посмотри `/ext/apps_data/meshcore_cfg/logs/<дата-время>/rx_log.csv`.

Успех: в файле есть строки, в колонках `snr` и `rssi` — числа.

> 🚨 **Если `rx_log.csv` только с заголовком и без строк** — нода не отдаёт
> rx-логи. Почти всегда это не та прошивка. Перешей нашей UART-сборкой.

**✅ Чек-лист части 2:** кабели A и B собраны · ноды перезагружены после втыкания ·
обе ноды подключаются · Connect показывает модель и версию ·
радио выставлено и подтверждено галочками · пробный лог пишется на SD.

# ЧАСТЬ 3. В ПОЛЕ — сплав по реке 🛶

### Что в сумке

5 нод (подписанных) · **флиппер с картой памяти** · повербанки для нод · кабели ·
телефон (для трейсов, см. ниже) · скотч/стяжки · пароли репитеров ·
**гермомешок и зип-пакеты на всё железо** · оффлайн-карты, скачанные заранее.

### Как ты пишешь лог

Ноды воткнуты и **перезагружены после втыкания** — иначе они подняли своё радио
и в serial-режим не пойдут.

Дальше никаких команд: **Apps → GPIO → MeshCore Config → Logger → OK.**

{{svg:screen-logger}} | Экран логгера в работе. Счётчик пакетов растёт — значит нода слышит эфир.

На экране: имя ноды, счётчик принятых пакетов, последние SNR и RSSI, имя папки сессии.
Счётчик растёт — всё пишется.

> 💡 **Back останавливает логгер и закрывает файлы.** Не выдёргивай питание
> посреди сессии просто так — хотя каждая строка и сбрасывается на карту сразу,
> так что даже в худшем случае ты потеряешь максимум одну.

**Главный бонус этой схемы:** обе ноды слушают одновременно, в одной точке. В CSV
строки помечены колонками `node` / `role` / `hw`, так что V4 и T114 в одном и том же
месте сравниваются напрямую — раньше такого сравнения получить было нельзя.

**Метки точек.** На экране логгера снизу кнопка **Mark**. Нажатие **OK** ставит
в `events.csv` строку с типом `mark`, номером и временем; счётчик меток тут же
меняется на экране, так что видно, что нажатие засчиталось.

> 💡 Номер метки называй вслух или записывай — потом по нему находишь строку в
> `events.csv`, а по времени подтягиваешь к ней SNR из `rx_log.csv`.

### Точечные трейсы — здесь нужен телефон

> 🚨 **Честно: флиппер этого не умеет.** Trace Path — пер-хоп SNR по каждому прыжку —
> это фича мобильного приложения MeshCore, и в companion-протоколе её нет в том виде,
> в каком её показывает приложение.
>
> В ключевых точках: **вынь провод из ноды, перезагрузи её** — она поднимет BLE, —
> цепляйся телефоном и делай **Tools → Trace Path** до каждой фиксированной ноды.
> Закончил — воткни провод обратно и снова перезагрузи ноду.
>
> Это единственный шаг, где телефон всё ещё обязателен.

### Спектр флиппером

Выйди из приложения (**Back** до главного меню флиппера) → **Sub-GHz → Frequency
Analyzer**, крути вокруг 868. На реке смотреть особо нечего — эфир чистый, — но
**на стоянке рядом с генератором или другой группой** проверить стоит. LoRa он не
декодирует, но «есть тут помеха и насколько мощная» показывает честно. Скриншотишь.

## Что потом с данными

**CSV с флиппера совместимы с `meshlog.py`** — колонки те же, в том же порядке,
просто в конце строки добавлены `node,role,hw`. Значит твои логи и логи Марка
(он пишет с телефона) сводятся вместе ровно как раньше.

- `rx_log.csv` — SNR/RSSI каждого услышанного пакета + координаты → **карта покрытия**
  (закинуть в Google Earth / My Maps по колонкам `lat`/`lon`).
- `ping.csv` — успех/промах и RTT по каждой ноде → **процент потерь и задержка**.
- `telemetry.csv` — батарея, шумовой пол, счётчики → **кривая разряда на день**.
- `events.csv` — advert-ы, сообщения, твои метки точек.

> 💡 Координаты флиппер берёт **из ноды**, своего GPS у него нет. Нет позиции
> в ноде — колонки `lat`/`lon`/`acc` останутся пустыми, всё остальное пишется.

# 🪤 ГРАБЛИ — все кочки, на которые мы уже наступили

Ищи по симптому.

## Флиппер и ноды по проводу

?? Нода молчит: «No reply from the node»
    Три причины, по убыванию частоты:

    1. **Перепутаны TX и RX.** Выход одной стороны идёт во вход другой:
       флиппер 13 → нода RX, нода TX → флиппер 14.
    2. **Не та прошивка.** Стоковый companion отдаёт протокол в USB-CDC или BLE.
       Нужна наша UART-сборка с [hleserg.github.io/meshcore-node-fw](https://hleserg.github.io/meshcore-node-fw).
    3. **Не тот пин.** На T114 нумерация на шелкографии не совпадает с нумерацией
       GPIO в Arduino. Сверяйся с таблицей распайки, а не с надписью на плате.

?? T114 молчит, а V4 на том же проводе работает
    На nRF52 в стоке консоль уходит только в USB-CDC — аппаратный UART не задействован.
    → Нужна прошивка с роутом на `Serial1`. Это и есть наша UART-сборка.

?? На экране «Noise on the line» или в Serial log мусор
    До флиппера что-то доходит, но это не кадры.
    → Не тот бод (нужен 115200) либо всё-таки перепутаны TX/RX.

?? `rx_log.csv` есть, но в нём только заголовок
    Нода не отдаёт rx-логи в компаньон.
    → Перешей ноду нашей UART-сборкой посвежее.

?? Приложение пишет «Cannot take the USART»
    Порт занят другим приложением флиппера.
    → Выйди из всех приложений и зайди в MeshCore Config заново.

?? Логи не появляются на карте вообще
    → Проверь, что SD-карта вставлена и смонтирована: на флиппере
      **Settings → Storage** должен показывать SD.

## Ноды и радио

?? Ноды не видят друг друга, хотя всё включено
    99% случаев — **разные радио-настройки**. freq/BW/SF/CR должны совпадать
    на всех пяти до последней цифры.
    → Сверь `meshcli -s COMx infos` на каждой. Или зайди в **Connect** на флиппере —
      он показывает текущее радио ноды прямо на экране.

?? На столе связь «глючит», а в поле всё нормально
    Пять передатчиков по 20+ dBm в 20 см друг от друга перегружают приёмники.
    → При настройке разноси ноды на 2–3 метра.

?? `set radio ...` ругается на синтаксис
    Синтаксис `set` слегка разнится между версиями прошивки.
    → Набери `set help` в CLI ноды. Или примени пресет с флиппера — там поля,
      ошибиться негде.

## Windows (настройка нод дома)

?? `pipx : The term 'pipx' is not recognized...`
    `pipx` — отдельная утилита, в Python не входит. **Она и не нужна.**
    → `py -m pip install --user meshcore-cli`

?? `meshcli` не находится после установки
    Папка `Scripts` не в PATH — классика Windows. Не воюй с PATH.
    → Запускай через модуль: `py -m meshcore_cli <аргументы>`

?? Не знаешь, какой COM-порт у ноды
    → `meshcli -l` покажет все serial- и BLE-устройства.
    → V4 не виден — доставь драйвер **CP210x**. T114 виден сам.

?? Странные ошибки кодировки, путь вида `C:\Users\й`
    Кириллица в домашней папке.
    → `py -m venv C:\mesh\venv` → `C:\mesh\venv\Scripts\activate` → ставить туда.

## Мини-словарик

- **SNR** (dB) — насколько сигнал торчит над шумом. **Главная метрика.** Для LoRa
  минус — норма, она вытягивает сигнал из-под шума примерно до −20 dB.
- **RSSI** (dBm) — просто громкость сигнала. Ближе к нулю = сильнее.
- **RTT** — время «туда-обратно» до подтверждения. **ACK** — само подтверждение.
- **advert** — маячок, которым нода объявляет о себе.
- **Trace Path** — MeshCore-аналог traceroute: гоняет тест-пакет по маршруту и
  возвращает SNR и время по каждому прыжку. **Только в мобильном приложении.**
- **Companion / Repeater** — роли: клиент (к нему цепляется флиппер или телефон) /
  ретранслятор.
- **SF / BW / CR** — крутилки модема LoRa (дальность ↔ скорость). На всех нодах
  сети должны совпадать.
- **Хоп** — прыжок через промежуточную ноду. Мультихоп = цепочка ретрансляторов.
- **ISM** — диапазоны для связи без лицензии (868 и 433 МГц).
- **USART / LPUART** — два аппаратных последовательных порта флиппера. Пины
  13/14 и 15/16 соответственно.
