# Тест меша MeshCore перед фестом — пошаговая инструкция 🐢

**Три части: что делать дома в нодах → что делать дома в телефонах → что делать в поле.**
Ноута в поле нет, всё крутится с двух телефонов.

---

## Расклад железа (5 нод)

| # | Нода | Роль | Прошивка | GPS+экран | Кто рулит |
|---|------|------|----------|-----------|-----------|
| 1 | **V4** (ESP32-S3) | 🎒 **ROVER-S** — главный логгер | `companion_radio_wifi` | ✅ | Телефон №1 (Termux, по WiFi) |
| 2 | **T114** | 🎒 **POCKET-S** — второй ходок | `companion` (BLE) | ✅ | Телефон №2 (приложение, по BLE) |
| 3 | **T114** | 🏠 **BASE** — якорь, высоко | `repeater` | — | по радио / USB |
| 4 | **T114** | 📡 **REP1** — скаут-ретранслятор | `repeater` | ✅ | по радио / USB |
| 5 | **T114** | 📡 **REP2** — ретранслятор | `repeater` | — | по радио / USB |

**Почему именно так, а не наоборот:** V4 — единственная нода с WiFi. Только через WiFi
телефон может отдать данные в Python-логгер (по TCP). У T114 (nRF52840) WiFi нет вообще,
только BLE — а **Termux в Android не умеет BLE** (нет доступа к системному Bluetooth-стеку).
Поэтому автологгер жёстко привязан к V4, а T114-ровер работает через приложение.

GPS ставим туда, где он реально нужен: на обоих роверов (позиция для каждой точки замера)
и на REP1 — его ты будешь переставлять, и удобно, когда он сам сообщает, где стоит.
База стоит на известной точке, ей координаты пропишем руками.

# ЧАСТЬ 1. ДОМА — В НОДАХ 🔧

Делается один раз, за столом, с компьютера по USB. Всё то же самое можно и через
приложение с телефона, просто с пятью нодами по USB быстрее.

> ⚠️ **Разнеси ноды на 2–3 метра, пока настраиваешь.** Пять передатчиков по 20+ dBm
> в 20 см друг от друга перегружают приёмники, и ты словишь странные «глюки связи»,
> которых в поле не будет.

### 1.0 Подготовка: CLI на компьютере (Windows)

`pipx` ставить **не надо** — обычного `pip` хватает.

```powershell
py --version                              # есть Python? нет - winget install Python.Python.3.12
py -m pip install --user meshcore-cli
meshcli -v                                # проверка
```

Если `meshcli` не находится — это папка `Scripts` не попала в PATH (классика Windows).
Не воюй, запускай через модуль, работает идентично:
```powershell
py -m meshcore_cli -v
```
**Дальше в гайде пиши `meshcli`, но если PATH не настроен — везде подставляй
`py -m meshcore_cli`.**

**Ищем COM-порт ноды** — в диспетчер устройств лезть не нужно, у CLI есть своя команда:
```powershell
meshcli -l          # покажет и serial-, и BLE-устройства
```
Порты на Windows называются `COM3`, `COM5` и т.п. (в линуксовых гайдах это `/dev/ttyACM0`).

- V4 не появился в списке → доставь драйвер **CP210x** (у ESP32-плат обычно он).
- T114 на nRF52840 определяется сам, драйвер не нужен.

> ⚠️ Если домашняя папка с кириллицей (`C:\Users\й`) и полезут странные ошибки
> кодировки — не мучайся, сделай venv по латинскому пути:
> `py -m venv C:\mesh\venv`, затем `C:\mesh\venv\Scripts\activate`, и ставь туда.

### 1.1 Прошить все пять
Веб-флешер: **flasher.meshcore.io** — прямо из браузера, компилятор не нужен.
- V4 → **`companion_radio_wifi`**
- T114 (ровер) → **`companion`** (BLE-вариант)
- 3× T114 → **`repeater`**

### 1.2 Поставить одинаковое радио на ВСЕХ ПЯТИ
Это самый частый и самый обидный косяк: если freq/BW/SF/CR не совпадают — ноды
физически не слышат друг друга, и ты полдня ищешь несуществующую поломку.

Рабочий вариант под 868 МГц ISM:
```
freq = 869.525 МГц   BW = 250 кГц   SF = 11   CR = 5
```
(433 МГц — тоже ISM, лучше пробивает препятствия, но канал уже и шумнее. Главное —
одинаково на всех пяти.)

Через CLI (компаньоны — V4 и T114-ровер), подставь свой COM-порт:
```powershell
meshcli -s COM5 set radio 869.525,250,11,5
meshcli -s COM5 set name ROVER-S
```
Репитеры настраиваются своим CLI по USB: `meshcli -s COM5 -r`, дальше `set ...`.

> 💡 Синтаксис `set` немного разнится между версиями прошивки. Не гадай — набери
> **`set help`** в CLI ноды и посмотри список. Либо выстави радио в мобильном
> приложении: там просто поля, ошибиться негде.

### 1.3 Дать имена
Твои пять: `BASE`, `REP1`, `REP2`, `ROVER-S`, `POCKET-S`.
Ровно этими именами ты потом будешь пинговать, и они попадут в CSV.
Наклей стикеры на корпуса — в поле спасает.

> 🤝 **Мы объединяемся с Марком, в сети будет 8 нод.** Он настраивает свои три
> по отдельной инструкции: `ROVER-M`, `POCKET-M` и `REP3` (его репитер уходит
> в общую инфраструктуру как резерв под «закрытие дыр»).
> **Радио-настройки обязаны совпадать** у обоих комплектов до последней цифры —
> сверьтесь заранее. Контактами тоже обменяйтесь дома: `export_contact` выдаёт
> URI, его можно переслать в телеграм.

### 1.4 Сменить пароли на трёх репитерах
У репитера есть админ-CLI по радио, и дефолтный пароль общеизвестен. Ставь свой —
иначе любой желающий на фесте перенастроит тебе сеть. Запиши пароли в телефон.

### 1.5 Синхронизировать часы
```powershell
meshcli -s COM5 clock sync
```
Без этого метки времени в логах разъедутся и сопоставить данные двух роверов не выйдет.

### 1.6 Прописать координаты базе
BASE поставишь в известную точку — вбей её широту/долготу заранее (`set lat`, `set lon`).
Тогда на карте она встанет ровно там, где стоит.

### 1.7 Познакомить ноды между собой
Включи все пять рядом (помним про 2–3 метра) и дай каждой отправить advert («маячок»).
Проверь, что в списке контактов каждая видит остальных:
```powershell
meshcli -s COM5 contacts
```
**Зачем:** в поле ты не хочешь стоять и ждать, пока ноды обнаружат друг друга.
Пусть контакты уже лежат в памяти.

### 1.8 Зарядить и собрать
Всё под завязку + повербанки. **V4 в режиме WiFi-точки жрёт заметно больше остальных** —
ему повербанк обязателен, а не «на всякий случай».

**✅ Чек-лист части 1:** прошито 5/5 · радио одинаковое · имена заданы · пароли сменены ·
часы синхронизированы · координаты базы вбиты · контакты видны · всё заряжено.

# ЧАСТЬ 2. ДОМА — В ТЕЛЕФОНАХ 📱

## Телефон №1 — с V4, главный логгер (Termux)

### 2.1 Termux — ставить с **F-Droid**, не с Google Play
Версия в Play исторически отставала и ломалась. Бери с F-Droid.
Плюс отдельно поставь приложение **Termux:API** (оттуда же).

### 2.2 Зеркало и ПОЛНОЕ обновление — в этом порядке!

**Сначала выбери быстрое зеркало**, иначе будешь качать 86 МБ на скорости 150 кБ/с:
```bash
termux-change-repo
```
Стрелками: `Mirror group` → выбери регион (для РФ живые — `repository.su`,
`mirror.mephi.ru`) → OK.

**Теперь обновляем ВСЁ и только потом ставим:**
```bash
pkg upgrade                 # ← именно upgrade, не update!
pkg install python termux-api clang
```

> 🚨 **Самая коварная кочка. `pkg update` ≠ `pkg upgrade`.**
> `update` только скачивает списки пакетов, **он ничего не обновляет**. Если после
> него сделать `pkg install python`, ты получишь новый Python поверх старых
> библиотек — а Termux это rolling-release, там так нельзя.
>
> Как выглядит поломка:
> ```
> ImportError: dlopen failed: cannot locate symbol "XML_SetHashSalt16Bytes"
> referenced by ".../pyexpat.cpython-314-....so"
> ```
> Это новый Python 3.14 просит символ у старой `libexpat`. Лечится тем же
> `pkg upgrade` — дообновить всё остальное, соглашаясь на все вопросы.

### 2.3 Если `pip` ругается `bad interpreter`

```
bash: .../bin/pip: .../bin/python3.13: bad interpreter: No such file or directory
```
Termux поднял Python на мажорную версию (3.13 → 3.14), а скрипт `pip` остался
от старой и указывает на несуществующий интерпретатор. Ставим pip заново:
```bash
python -m ensurepip --upgrade
python -m pip --version
```
**Дальше везде вызывай `python -m pip`, а не голый `pip`** — тогда эта болячка
не вернётся после следующего обновления.

> Если `ensurepip` падает с тем же `pyexpat`/`dlopen` — значит ты ещё не доделал
> `pkg upgrade` из пункта 2.2. Вернись и доделай.

### 2.4 Ставим библиотеку — сразу через `--no-deps`

Обычный `pip install meshcore` на Termux **упадёт**, и это не случайность:
он тянет `bleak` (Bluetooth), а `bleak` на Linux/Android требует `dbus-fast`,
который надо компилировать с dbus-заголовками. На Termux это не собирается.

**Нам `bleak` не нужен — мы работаем по TCP.** Проверено: без него библиотека
живёт прекрасно. Поэтому ставим так:
```bash
python -m pip install meshcore --no-deps
python -m pip install pycayennelpp pycryptodome pyserial-asyncio-fast
```

Из трёх зависимостей две — чистый Python, а **`pycryptodome` собирается из
исходников** (для этого и ставили `clang`). Пара минут на телефоне — норма.
Он обязателен: без него `meshcore` не импортируется совсем.

Проверка, что всё живо:
```bash
python -c "from meshcore import MeshCore; print('ok', hasattr(MeshCore,'create_tcp'))"
```
Должно напечатать `ok True`.

### 2.5 Закинуть `meshlog.py` в Termux

Скрипт должен лежать в **домашней папке Termux** (`~`), а не в общей памяти телефона.

**Шаг 1. Дать Termux доступ к памяти** (один раз, Android спросит разрешение):
```bash
termux-setup-storage
```

**Шаг 2. Найти скачанный файл.** Он может быть где угодно — зависит от того, откуда
качал и какая версия Android. Не гадай, просто найди:
```bash
find ~/storage/shared -name "meshlog.py" 2>/dev/null
```

Типичные места:
| Откуда качал | Путь |
|---|---|
| Браузер / «Сохранить в Загрузки» | `~/storage/downloads/meshlog.py` |
| **Telegram** (Android 11+) | `~/storage/shared/Android/media/org.telegram.messenger/Telegram/Telegram Documents/` |
| **Telegram** (старые версии) | `~/storage/shared/Telegram/Telegram Documents/` |

**Шаг 3. Скопировать к себе.** Подставь путь, который выдал `find`
(кавычки обязательны — в пути Telegram есть пробелы):
```bash
cp "$(find ~/storage/shared -name 'meshlog.py' 2>/dev/null | head -1)" ~/
ls -l ~/meshlog.py
```

> Если `find` выдал **несколько** путей (скачивал дважды) — однострочник возьмёт
> первый попавшийся, а он может оказаться старой копией. В этом случае копируй
> явно, указав нужный путь в кавычках:
> `cp "~/storage/shared/.../Telegram Documents/meshlog.py" ~/`

**Запасной вариант — через буфер обмена.** Если с файлом совсем беда: открой
`meshlog.py` как текст, скопируй целиком, потом в Termux:
```bash
termux-clipboard-get > ~/meshlog.py
```

**Шаг 4. Проверить:**
```bash
cd ~
python meshlog.py --help
```
Показал справку — готово.

> 📁 **Запускай логгер всегда из домашней папки** (`cd ~`), не из `~/storage/...`.
> Скрипт пишет CSV в `./meshtest` относительно текущего каталога — во внутренней
> памяти это быстро и без сюрпризов с правами доступа.
> Забрать логи в телефон после теста:
> ```bash
> cp -r ~/meshtest ~/storage/downloads/
> ```

### 2.6 Разрешить геолокацию
```bash
termux-location
```
Первый раз Android спросит разрешение — дай. Должен вернуться JSON с координатами.
**Если пропустить — в CSV не будет позиций, и карту покрытия не построить.**

### 2.7 Победить засыпание и мобильный интернет
- `termux-wake-lock` — иначе Android усыпит Termux в кармане и лог оборвётся.
- В настройках батареи сними с Termux все ограничения энергосбережения.
- **Отключи мобильные данные на время прогона.** Точка доступа V4 без интернета,
  и Android может решить «этот WiFi бесполезный» и увести трафик в LTE — тогда
  логгер не достучится до ноды.

### 2.8 Оффлайн-карты
OsmAnd или Organic Maps — скачай район феста заранее. В поле интернета не будет.

### 2.9 Генеральная репетиция дома (не пропускай!)
Включи V4, подключись телефоном к его WiFi, посмотри **IP на экранчике V4** и запусти:
```bash
python meshlog.py --tcp 192.168.4.1:5000 --ping BASE
```
Убедись, что: подключается · в консоли бегут строки `ping BASE #N: OK ... ms` ·
в папке `./meshtest` появились CSV и они растут.

> Если порт 5000 не отвечает — попробуй 4000. И бери IP именно с экрана ноды,
> не угадывай.

# ЧАСТЬ 3. В ПОЛЕ 🏕️

### Что в сумке
5 нод (подписанных) · 2 телефона · 2+ повербанка · кабели · Флиппер ·
скотч/стяжки/пакеты от дождя · записанные пароли репитеров.

---

### Шаг 1. Ставим BASE
Самая высокая доступная точка: крыша, мачта, дерево, борт фуры. **Высота решает
больше, чем мощность** — подъём на 3 метра часто даёт больше, чем любые пляски
с антенной. Включил → убедился, что живая (экран/advert дошёл до ровера).

### Шаг 2. Раскидываем REP1 и REP2
Ставь в кандидатные точки: между базой и дальним краем площадки, за препятствиями
(сцена, холм, постройки) — там, где прямой луч до базы не проходит.
REP1 с GPS+экраном — его удобно переставлять и сверять позицию по экрану.

### Шаг 3. Запускаем логгер (телефон №1)
Подключаемся к WiFi ноды V4, смотрим IP на её экранчике, и:
```bash
termux-wake-lock
python meshlog.py --tcp 192.168.4.1:5000 --ping BASE,REP1,REP2 --interval 15
```
Теперь он **по кругу** пингует все три фиксированные ноды и параллельно пишет
всё, что слышит. Дальше просто ходишь — данные копятся сами.

### Шаг 4. Прогон по площадке
Идёшь и **не спешишь** — стой на каждой точке 30–60 секунд, чтобы набралось
несколько пингов до каждой ноды.

Куда обязательно дойти:
- по периметру площадки, включая самый дальний угол;
- где будут сцены и лагерь;
- в низины и за постройки (это worst case);
- на въезд/парковку.

**Ставить метки в логе без рук:** отправь с телефона №2 сообщение в канал вроде
`TOCHKA scene2`. Оно прилетит на ровер №1 и запишется в `events.csv` с временем —
потом сопоставишь с координатами и поймёшь, где ты был.

### Шаг 5. Точечные трейсы (телефон №2)
В ключевых точках: **Tools → Trace Path** до каждой фиксированной ноды.
Смотришь **SNR** и время отклика по каждому хопу — сразу видно, какое звено слабое
и реально ли работает мультихоп через репитеры.

### Шаг 6. Спектр Флиппером
Sub-GHz → **Frequency Analyzer**, крутим вокруг 868 (или 433). Смотрим в местах
скопления техники: генераторы, звук, LED-экраны, Starlink. LoRa он не декодирует,
но «есть тут помеха и насколько мощная» показывает честно. Скриншотишь.

### Шаг 7. Сворачиваемся
`Ctrl-C` в Termux (CSV уже сохранены, скрипт пишет на диск сразу — даже если
телефон вырубится, данные на месте). Скопируй папку `meshtest` из Termux в
обычную память телефона.

## Критерии приёмки — что считать «сеть готова»

| Метрика | Где смотреть | Порог |
|---|---|---|
| SNR до ближайшей фикс-ноды в любой точке | `rx_log.csv`, трейсы | **≥ +5 dB** |
| Потери пингов | `ping.csv`, колонка `ok` | **< 5%** |
| Мультихоп работает | Trace Path через REP | путь реально строится |
| Батарея базы/репитеров | `telemetry.csv` | хватает на весь день |
| Ошибки приёма | `telemetry.csv`, `recv_errors` | не растут лавиной |

> 🚨 **Главная поправка на реальность.** Ты тестируешь пустую площадку, а фест —
> это тысячи человеческих тел, и они **очень хорошо поглощают 868 МГц**. Плюс
> генераторы, звук, толпа с телефонами. Поэтому закладывай запас: если на пустом
> поле SNR впритык 0 dB — на фесте эта связь ляжет. **Целься в +5…+10 dB запаса**
> либо ставь ещё один репитер.

## Что потом с данными

# 🪤 ГРАБЛИ — все кочки, на которые мы уже наступили

Ищи по симптому. Всё ниже — реально случилось, не теория.

## Windows (настройка нод дома)

**`pipx : The term 'pipx' is not recognized...`**
`pipx` — отдельная утилита, в Python не входит. **Она и не нужна.**
→ `py -m pip install --user meshcore-cli`

**`meshcli` не находится после установки**
Папка `Scripts` не в PATH — классика Windows. Не воюй с PATH.
→ Запускай через модуль: `py -m meshcore_cli <аргументы>`

**Не знаешь, какой COM-порт у ноды**
В диспетчер устройств лезть не надо.
→ `meshcli -l` покажет все serial- и BLE-устройства.
→ V4 не виден — доставь драйвер **CP210x**. T114 (nRF52840) виден сам.

**Странные ошибки кодировки, путь вида `C:\Users\й`**
Кириллица в домашней папке. Обычно проносит (всё ставится колёсами), но если нет:
→ `py -m venv C:\mesh\venv` → `C:\mesh\venv\Scripts\activate` → ставить туда.

## Termux (телефон №1)

**`ImportError: dlopen failed: cannot locate symbol "XML_SetHashSalt16Bytes"`**
(или любой другой `cannot locate symbol` / `dlopen failed`)
**Причина: `pkg update` ≠ `pkg upgrade`.** `update` только качает списки пакетов,
ничего не обновляя. Ты поставил новый Python поверх старых библиотек, а Termux —
rolling-release, там так нельзя. Новый `pyexpat` просит символ у старой `libexpat`.
→ `pkg upgrade` — дообновить всё, соглашаясь на все вопросы.

**`bash: .../pip: .../python3.13: bad interpreter: No such file`**
Termux поднял Python 3.13 → 3.14, скрипт `pip` остался от старого.
→ `python -m ensurepip --upgrade`
→ **дальше всегда `python -m pip`, не голый `pip`.**
→ Если `ensurepip` падает с `dlopen` — сначала доделай `pkg upgrade`.

**`pip install meshcore` падает на `bleak` / `dbus-fast`**
Не случайность, а гарантия: `bleak` на Linux (Android — это Linux) требует
`dbus-fast`, а тот компилируется с dbus-заголовками, которых на Termux нет.
**BLE нам тут не нужен — идём по TCP.**
→ `python -m pip install meshcore --no-deps`
→ `python -m pip install pycayennelpp pycryptodome pyserial-asyncio-fast`

**`ModuleNotFoundError: No module named 'Crypto'`**
`pycryptodome` **обязателен** — без него `meshcore` не импортируется вообще
(в отличие от `bleak`, без которого всё прекрасно живёт).
→ Ставь его, он единственный из трёх собирается из исходников (нужен `clang`).

**Всё качается мучительно медленно**
Выбралось далёкое зеркало.
→ `termux-change-repo` → Mirror group → свой регион.

**Лог обрывается через несколько минут**
Android усыпил Termux.
→ `termux-wake-lock` перед запуском + сними ограничения энергосбережения
  с Termux в настройках батареи.

**Красный `ERROR: ... meshcore 2.3.7 requires bleak, which is not installed`**
**Это НЕ ошибка, всё встало правильно.** Смотри строку ниже — там будет
`Successfully installed ...`. pip просто предупреждает, что в метаданных `meshcore`
заявлен `bleak`, а мы его намеренно пропустили. Игнорируй и иди дальше.
→ Проверка, что реально всё ок:
  `python -c "from meshcore import MeshCore; print('ok', hasattr(MeshCore,'create_tcp'))"`
  → должно быть `ok True`.

**Долгая сборка `pycryptodome` (несколько минут, куча вывода)**
Это норма — он единственный компилируется из исходников. На Python 3.14 /
Android aarch64 собирается штатно (проверено).
→ Просто дождись `Successfully built pycryptodome`.

**`python: can't open file 'meshlog.py'` / файл не находится**
Скрипт лежит не в домашней папке Termux, либо ты не в ней.
→ `termux-setup-storage` (один раз), затем найди файл:
  `find ~/storage/shared -name "meshlog.py" 2>/dev/null`
→ Telegram на Android 11+ кладёт сюда:
  `~/storage/shared/Android/media/org.telegram.messenger/Telegram/Telegram Documents/`
  (в пути есть пробелы — при копировании оборачивай в кавычки!)
→ Скопируй в `~` и запускай из `cd ~`, иначе CSV лягут не туда, куда ждёшь.

## Ноды и радио

**Ноды не видят друг друга, хотя всё включено**
99% случаев — **разные радио-настройки**. freq/BW/SF/CR должны совпадать
**на всех пяти** до последней цифры.
→ Сверь `meshcli -s COMx infos` на каждой.

**На столе связь «глючит», а в поле всё нормально**
Пять передатчиков по 20+ dBm в 20 см друг от друга перегружают приёмники.
→ При настройке разноси ноды на 2–3 метра.

**`set radio ...` ругается на синтаксис**
Синтаксис `set` слегка разнится между версиями прошивки.
→ Набери `set help` в CLI ноды. Или выстави радио в мобильном приложении —
  там поля, ошибиться негде.

## Логгер в поле

**Логгер не подключается к V4 по TCP**
→ Порт: пробуй `5000`, затем `4000`.
→ IP бери **с экранчика V4**, не угадывай.
→ **Отключи мобильные данные.** Точка доступа V4 без интернета, и Android
  уводит трафик в LTE — до ноды логгер не достучится.

**`ping ... MISS` на все цели подряд, хотя ноды рядом**
Контактов ещё нет в памяти ноды.
→ Дай нодам обменяться advert-ами (п. 1.7 — делается дома, до выезда).

**`rx_log.csv` пустой, а остальные CSV пишутся**
Прошивка ноды не отдаёт rx-логи в компаньон.
→ Обнови до свежей `companion_radio_*`. Уровень сигнала всё равно виден
  в `ping.csv` и через `meshcli trace`.

**В CSV нет координат (`lat`/`lon` пустые)**
Не выдано разрешение на геолокацию или не стоит `termux-api`.
→ Проверь `termux-location` — должен вернуть JSON с координатами.



- **SNR** (dB) — насколько сигнал торчит над шумом. **Главная метрика.** Для LoRa
  минус — норма, она вытягивает сигнал из-под шума примерно до −20 dB.
- **RSSI** (dBm) — просто громкость сигнала. Ближе к нулю = сильнее.
- **RTT** — время «туда-обратно» до подтверждения. **ACK** — само подтверждение.
- **advert** — маячок, которым нода объявляет о себе.
- **Trace Path** — MeshCore-аналог traceroute: гоняет тест-пакет по маршруту и
  возвращает SNR и время по каждому прыжку.
- **Companion / Repeater** — роли: клиент (к нему цепляется телефон) / ретранслятор.
- **SF / BW / CR** — крутилки модема LoRa (дальность ↔ скорость). На всех нодах
  сети должны совпадать.
- **Хоп** — прыжок через промежуточную ноду. Мультихоп = цепочка ретрансляторов.
- **ISM** — диапазоны для связи без лицензии (868 и 433 МГц).
