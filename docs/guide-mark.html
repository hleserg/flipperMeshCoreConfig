MeshCore: настройка и полевой тест — инструкция для Марка 🐢

Мы объединяем железо: в сети будет 8 нод. Твои три настраиваешь ты (эта инструкция), остальные пять — Сергей. Документ самодостаточный, ничего другого читать не надо.

Твои ноды и их роли

Твоя нодаИмя в сетиРольПрошивка
V4 (ESP32-S3)ROVER-M🎒 Автологгер — ходит с тобой, сам пишет данныеcompanion_radio_wifi
T114 №1POCKET-M🎒 Носимая «пользовательская» нодаcompanion (BLE)
T114 №2REP3📡 Ретранслятор — уезжает в общую инфраструктуруrepeater

Зачем тебе два ровера в кармане. У них разные задачи: V4 — это измерительный прибор (к нему цепляется телефон по WiFi, крутится Python-логгер, пишутся CSV). T114 — это то, что мы, собственно, тестируем: маленькая, экономичная, именно такая будет у людей на фесте. Заодно на ней меряем реальный расход батареи за день.

Вся сеть целиком (для понимания):

НодаЧьяРоль
BASEСергейякорь, ставится на высшую точку
REP1, REP2Сергейретрансляторы
REP3твояретранслятор / резерв под «закрытие дыр»
ROVER-S, POCKET-SСергейего роверы
ROVER-M, POCKET-Mтвоитвои роверы

ЧАСТЬ 1. ДОМА — НАСТРОЙКА ТВОИХ ТРЁХ НОД 🔧

Делается один раз, за столом, с компьютера по USB.

⚠️ Разнеси ноды на 2–3 метра, пока настраиваешь. Передатчики по 20+ dBm в 20 см друг от друга перегружают приёмники, и ты словишь «глюки связи», которых в реальности нет.

1.0 Поставить CLI на компьютер

pipx ставить не надо, хватает обычного pip.

Windows (PowerShell):

py --version                              # нет Python? → winget install Python.Python.3.12
py -m pip install --user meshcore-cli
meshcli -v

macOS / Linux:

python3 -m pip install --user meshcore-cli
meshcli -v

Если meshcli не находится — папка Scripts/bin не в PATH. Не воюй с PATH, запускай через модуль, работает идентично:

py -m meshcore_cli <аргументы>        # Windows
python3 -m meshcore_cli <аргументы>   # mac/Linux

Найти порт ноды — в диспетчер устройств лезть не нужно:

meshcli -l

Порт на Windows выглядит как COM3/COM5, на mac/Linux — /dev/tty.usbserial-* или /dev/ttyACM0. Дальше в инструкции пишу COM5 — подставляй свой.

  • V4 не появился в списке → доставь драйвер CP210x (у ESP32-плат обычно он).
  • T114 (nRF52840) определяется сам, драйвер не нужен.

1.1 Прошить все три

Веб-флешер: flasher.meshcore.io — прямо из браузера, компилятор не нужен.

НодаПрошивка
V4companion_radio_wifi ← именно WiFi-вариант, без него логгер не заработает
T114 №1companion (BLE-вариант)
T114 №2repeater

1.2 🚨 Радио — самое важное. Цифры менять нельзя

Если у твоих нод и у нод Сергея радио-настройки отличаются хоть на йоту — ноды физически не услышат друг друга, и мы полдня будем искать несуществующую поломку. Ставь ровно это, на все три:

freq = 869.525 МГц    BW = 250 кГц    SF = 11    CR = 5
meshcli -s COM5 set radio 869.525,250,11,5

💡 Синтаксис set слегка разнится между версиями прошивки. Не гадай — набери set help в CLI ноды и посмотри список. Либо выстави радио в мобильном приложении: там просто поля, ошибиться негде.

Проверить, что применилось:

meshcli -s COM5 infos

Сверь radio_freq, radio_bw, radio_sf, radio_cr с цифрами выше.

1.3 Дать имена

Ровно эти, они попадут в CSV и по ним мы будем сводить данные:

meshcli -s COM5 set name ROVER-M      # твой V4
meshcli -s COM5 set name POCKET-M     # твой T114 №1
meshcli -s COM5 set name REP3         # твой T114 №2

Наклей стикеры с именами на корпуса — в поле, когда нод восемь, это спасает.

1.4 Пароль на REP3

У репитера есть админ-CLI, доступный по радио, и дефолтный пароль общеизвестен. Поставь свой, иначе любой желающий на фесте перенастроит сеть. Репитер настраивается своим CLI: meshcli -s COM5 -r, дальше команды. Пароль запиши и передай Сергею — репитер уходит в общую инфраструктуру, управлять им может понадобиться обоим.

1.5 Синхронизировать часы

meshcli -s COM5 clock sync

На всех трёх. Это критично: мы будем сводить логи с двух телефонов, и если время разъедется — данные не склеятся.

1.6 Познакомить ноды и передать контакты Сергею

Включи все три рядом (помним про 2–3 метра), дай каждой отправить advert:

meshcli -s COM5 advert
meshcli -s COM5 contacts     # каждая должна видеть две других

Чтобы в поле не ждать обнаружения, перешли Сергею свои контакты заранее:

meshcli -s COM5 export_contact

Выдаст URI вида meshcore://... — кинь все три в телеграм. Он их импортирует (import_contact <URI> или сканом QR в приложении), а тебе пришлёт свои пять.

1.7 Зарядить

Всё под завязку + повербанк. V4 в режиме WiFi-точки жрёт заметно больше остальных — ему повербанк обязателен, а не «на всякий случай».

✅ Чек-лист части 1: прошито 3/3 · радио 869.525/250/11/5 на всех трёх · имена ROVER-M/POCKET-M/REP3 · пароль на REP3 задан и передан · часы синхронизированы · контакты выгружены и отправлены · всё заряжено.

ЧАСТЬ 2. ДОМА — НАСТРОЙКА ТЕЛЕФОНА 📱

Тебе нужны два приложения на одном телефоне: Termux (логгер для V4) и клиент MeshCore (для T114).

2A. Termux — логгер для ROVER-M

2.1 Ставить Termux с F-Droid, не с Google Play

Версия в Play исторически отставала и ломалась. Плюс отдельно поставь приложение Termux:API (оттуда же).

2.2 Зеркало и ПОЛНОЕ обновление — в этом порядке

Сначала выбери быстрое зеркало, иначе будешь качать 86 МБ на скорости 150 кБ/с:

termux-change-repo

Стрелками: Mirror group → свой регион → OK.

Теперь обновляем всё и только потом ставим:

pkg upgrade                 # ← именно upgrade, не update!
pkg install python termux-api clang

🚨 Кочка, на которую мы уже наступили. pkg update ≠ pkg upgrade. update только скачивает списки пакетов, ничего не обновляя. Если после него поставить python, получишь новый Python поверх старых библиотек — а Termux это rolling-release, там так нельзя. Выглядит так:

ImportError: dlopen failed: cannot locate symbol "XML_SetHashSalt16Bytes"

Лечится тем же pkg upgrade — дообновить всё, соглашаясь на все вопросы.

2.3 Если pip ругается bad interpreter

bash: .../bin/pip: .../bin/python3.13: bad interpreter: No such file or directory

Termux поднял Python на мажорную версию, а скрипт pip остался от старой.

python -m ensurepip --upgrade
python -m pip --version

Дальше везде вызывай python -m pip, а не голый pip. Если ensurepip падает с тем же dlopen — сначала доделай pkg upgrade.

2.4 Ставим библиотеку — сразу через --no-deps

Обычный pip install meshcore на Termux упадёт, и это закономерно: он тянет bleak (Bluetooth), а тот на Linux/Android требует dbus-fast, который надо компилировать с dbus-заголовками. На Termux это не собирается.

bleak нам не нужен — логгер работает по TCP. Проверено, без него всё живёт:

python -m pip install meshcore --no-deps
python -m pip install pycayennelpp pycryptodome pyserial-asyncio-fast

pycryptodome будет пару минут собираться из исходников — это норма, для того и ставили clang. Он обязателен: без него meshcore не импортируется вообще.

В конце pip напишет красным: ERROR: ... meshcore 2.3.7 requires bleak, which is not installed Это не ошибка. Ниже будет Successfully installed .... pip просто ворчит, что мы намеренно пропустили bleak. Игнорируй.

Проверка:

python -c "from meshcore import MeshCore; print('ok', hasattr(MeshCore,'create_tcp'))"

Должно быть ok True.

2.5 Закинуть meshlog.py в Termux

Скрипт должен лежать в домашней папке Termux (~), а не в общей памяти.

Дать доступ к памяти (один раз, Android спросит разрешение):

termux-setup-storage

Найти скачанный файл — не гадай, где он, просто найди:

find ~/storage/shared -name "meshlog.py" 2>/dev/null
Откуда качалГде искать
Браузер / «Сохранить в Загрузки»~/storage/downloads/
Telegram (Android 11+)~/storage/shared/Android/media/org.telegram.messenger/Telegram/Telegram Documents/
Telegram (старые версии)~/storage/shared/Telegram/Telegram Documents/

Скопировать к себе (кавычки обязательны — в пути Telegram есть пробелы):

cp "$(find ~/storage/shared -name 'meshlog.py' 2>/dev/null | head -1)" ~/
ls -l ~/meshlog.py

Запасной вариант — открой файл как текст, скопируй целиком, потом:

termux-clipboard-get > ~/meshlog.py

Проверить:

cd ~
python meshlog.py --help

📁 Запускай логгер всегда из домашней папки (cd ~). Скрипт пишет CSV в ./meshtest относительно текущего каталога. Забрать логи потом: cp -r ~/meshtest ~/storage/downloads/

2.6 Разрешить геолокацию

termux-location

Первый раз Android спросит разрешение — дай. Должен вернуться JSON с координатами. Если пропустить — в CSV не будет позиций, и карту покрытия не построить.

2.7 Победить засыпание и мобильный интернет

  • termux-wake-lock перед запуском — иначе Android усыпит Termux в кармане.
  • В настройках батареи сними с Termux все ограничения энергосбережения.
  • Отключи мобильные данные на время прогона. Точка доступа V4 без интернета,

и Android может увести трафик в LTE — тогда логгер не достучится до ноды.

2.8 Генеральная репетиция дома (не пропускай!)

Включи V4, подключись телефоном к его WiFi, посмотри IP на экранчике V4:

cd ~
python meshlog.py --tcp 192.168.4.1:5000 --ping REP3

Убедись, что подключается, в консоли бегут ping REP3 #N: OK ... ms, и в ~/meshtest появились CSV и растут.

Порт не отвечает — попробуй 4000. IP бери с экрана ноды, не угадывай.

2.10 Спарить T114 по Bluetooth

Сначала пара в системных настройках Android, потом коннект в приложении.

2.11 Включить GPS в самом приложении

ЧАСТЬ 3. В ПОЛЕ — совместный прогон 🏕️

Работаем вдвоём, и это главное преимущество: два логгера пишут одновременно.

Что взять

Три ноды (подписанные) · телефон · повербанк · кабели · пароль от REP3.

---

Фаза 0. Сверка на месте (10 минут)

  1. Сергей ставит BASE на высшую точку, REP1 и REP2 — по кандидатным точкам.

REP3 (твой) пока держим в резерве — он понадобится в Фазе 5.

  1. Оба включают роверов, дают advert.
  2. Перекличка: оба проверяют, что видят все ноды сети. Не видно чью-то —

разбираемся сейчас, а не через два часа.

  1. Сверяем часы на телефонах и записываем время старта.

Фаза 1. Параллельный обход

Расходитесь в противоположные стороны по периметру площадки. Оба запускают:

termux-wake-lock
cd ~
python meshlog.py --tcp 192.168.4.1:5000 --ping BASE,REP1,REP2 --interval 15

Идёте не спеша: на каждой точке стой 30–60 секунд, чтобы набралось несколько пингов до каждой ноды. Покрытие собирается вдвое быстрее, чем в одиночку.

Куда дойти обязательно: периметр целиком, самый дальний угол, места будущих сцен и лагеря, низины и «тени» за постройками, въезд/парковка.

Метки в логе без рук: отправь с POCKET-M сообщение в канал вроде TOCHKA vezd. Оно прилетит на оба ровера и запишется в events.csv с временем — потом сопоставишь с координатами.

Фаза 2. Опорная точка + ходок

Один встаёт на известную точку и стоит, второй ходит. Зачем: если сигнал просел, надо понимать — это позиция ухудшилась или канал в целом деградировал (погода, помеха, толпа). Неподвижный ровер даёт эталон для сравнения. Потом поменяйтесь ролями.

Фаза 3. Ровер ↔ ровер — главный сценарий феста

На фесте люди будут переписываться друг с другом, а не с базой. Проверяем именно это:

  1. Расходитесь на максимальное расстояние.
  2. С POCKET-M пишешь на POCKET-S, ждёшь доставки.
  3. В приложении Tools → Trace Path до POCKET-S — смотришь путь, SNR по

каждому хопу и задержку.

  1. Пробуешь позиции, где прямой связи заведомо нет — проверяем, что меш

действительно роутит через репитеры, а не «работает, пока видим друг друга».

Фаза 4. Точечные трейсы

В ключевых точках с приложения: Tools → Trace Path до BASE, REP1, REP2. SNR — главное число. Сразу видно, какое звено слабое.

Фаза 5. Закрыть дыру — вот зачем нужен REP3

Из данных Фаз 1–3 находим мёртвую зону. Ставим туда (или на точку, которая её перекрывает) твой REP3, включаем, ждём advert — и проходим этот кусок заново с логгером.

Смысл: это не «поставили и надеемся», а нормальный эксперимент — замер → вмешательство → повторный замер. Если дыра закрылась, цифры покажут. Если нет — двигаем REP3 и повторяем.

Фаза 6. Сворачиваемся

  • Ctrl-C в Termux. CSV пишутся на диск сразу, так что даже если телефон

вырубится — данные на месте.

  • Скопируй логи в память телефона: cp -r ~/meshtest ~/storage/downloads/
  • Переименуй папку в meshtest-mark и отправь Сергею — иначе при сведении

двух комплектов файлы перепутаются.

Критерии приёмки — что считать «сеть готова»

МетрикаГде смотретьПорог
SNR до ближайшей фикс-ноды в любой точкеrx_log.csv, трейсы≥ +5 dB
Потери пинговping.csv, колонка ok< 5%
Ровер↔ровер работает через мешФаза 3сообщения доходят
Батарея POCKET-Mtelemetry.csvхватает на весь день
Ошибки приёмаtelemetry.csv, recv_errorsне растут лавиной

🚨 Главная поправка на реальность. Мы тестируем пустую площадку, а фест — это тысячи человеческих тел, и они очень хорошо поглощают 868 МГц. Плюс генераторы, звук, толпа с телефонами. Если на пустом поле SNR впритык 0 dB — на фесте эта связь ляжет. Целься в запас +5…+10 dB либо ставь ещё репитер.

Мини-словарик

  • SNR (dB) — насколько сигнал торчит над шумом. Главная метрика качества.

Для LoRa минус — норма, она вытягивает сигнал из-под шума примерно до −20 dB.

  • RSSI (dBm) — просто громкость сигнала. Ближе к нулю = сильнее.
  • RTT — время «туда-обратно» до подтверждения. ACK — само подтверждение.
  • advert — маячок, которым нода объявляет о себе в сети.
  • Trace Path — MeshCore-аналог traceroute: гоняет тест-пакет по маршруту и

возвращает SNR и время по каждому прыжку.

  • Companion / Repeater — роли: клиент (к нему цепляется телефон) / ретранслятор.
  • SF / BW / CR — крутилки модема LoRa (дальность ↔ скорость). У всех нод сети

должны совпадать.

  • Хоп — прыжок через промежуточную ноду. Мультихоп = цепочка ретрансляторов.
  • ISM — диапазоны для связи без лицензии (868 и 433 МГц).
